# Spectra Security Assessment Tool

---

## Project Vision: The "Spectra" Hacking Panel

Spectra is an advanced, AI-driven security assessment platform designed as a **comprehensive hacking panel**. Unlike traditional scanners, Spectra operates as a **Multi-Agent System (MAS)**, orchestrating a suite of industry-standard tools through a modern, real-time web interface.

**Core Philosophy:**

* **Human-in-the-Loop (HITL):** AI suggests and executes, but the user has full visibility and control via a "Glass Cockpit" UI.
* **Confidence via Consensus:** Utilizing the **MAKER Framework**, multiple small AI models vote on decisions to eliminate hallucinations.
* **Live Operations:** Real-time feedback via WebSockets—watch the terminal output as it happens.

---

## 1. AI Architecture: The MAKER Framework

We are moving away from a monolithic "Decision Chain" to the **MAKER (Maximal Agentic decomposition, K-threshold Error mitigation, and Red-flagging)** architecture. This allows us to use smaller, faster models (e.g., 3B-8B parameters) with higher reliability than a single large model.

### The Swarm Structure

#### A. Recon Swarm (Discovery)

* **Scope Agent:** Parses user input to define strict boundaries (CIDR, domains).
* **Tool Selector:** Analyzes the target service (e.g., "Apache 2.4 on port 80") and votes on the best tool (Nikto vs. Ffuf).
* **Parameter Tuner:** Adjusts tool flags (e.g., `-T4` vs `-T2`) based on stealth requirements.

#### B. Analysis Swarm (Consensus Engine)

* **Parser Agents:** Specialized micro-models that only know how to read specific outputs (e.g., one agent just for Nmap XML, another for Nikto text).
* **Voting System (K-Threshold):**
  * *Scenario:* Is this SQL Injection real?
  * *Action:* Three distinct model instances analyze the request/response.
  * *Threshold:* If 2/3 agree, it is flagged as "Verified". If not, it is "Potential".
* **Risk Scorer:** Assigns contextual risk scores (not just CVSS) based on asset criticality and exploitability.

#### C. Exploitation Swarm (Red Team)

* **Payload Crafter:** Generates context-specific payloads (e.g., bypassing a specific WAF rule) using SearchSploit and CVE data.
* **Exploit Verifier:** Monitors the outcome of an exploit attempt to confirm success (e.g., checking for a reverse shell connection or specific HTTP response) to provide Proof-of-Exploit.
* **Safety Supervisor (Red-Flagging):** A dedicated agent that intercepts commands *before* execution. It blocks dangerous actions (e.g., `rm -rf`, `DROP TABLE`) regardless of the Crafter's intent.

#### D. Strategic Swarm (Steering)

* **Mission Controller:** Accepts high-level user directives (e.g., "Establish C2", "Exfiltrate Data", "Passive Recon Only"). Has **Override Authority** over the Voting System to force specific paths.
* **Auto-Pilot:** If no directive is given, it defaults to the "Standard Assessment" profile (Discovery -> Analysis -> Exploitation -> Reporting).

## 2. Tool Suite & Modules

The platform is divided into modular capabilities. Each module contains a curated set of tools selected for their effectiveness and compatibility with AI orchestration.

### Module A: Scanning & Enumeration

* **Naabu:** Fast port scanner (Written in Go, integrates with Nuclei).
* **Nuclei:** Template-based vulnerability scanning (High speed).
* **Amass:** Advanced subdomain enumeration and mapping.
* **Ffuf:** Fast web fuzzer (Directory/VHost discovery).
* **Feroxbuster:** Recursive content discovery (Rust-based, faster alternative to Dirbuster).
* **Katana:** Next-generation crawler and spider.
* **Nikto:** Web server scanner (Legacy but reliable for misconfigs).
* **Masscan:** Internet-scale port scanner (Strictly rate-limited by Safety Supervisor).

### Module B: Secrets & API Security

* **TruffleHog:** Scans for leaked credentials and secrets in code/responses.
* **Kiterunner:** API endpoint discovery and scanning.
* **Gitleaks:** Detects hardcoded secrets in git repositories.

### Module C: Exploit Development

* **Searchsploit:** Offline exploit database search.
* **MSFpc:** Metasploit Payload Creator (Quick payload generation).
* **MSFvenom:** Advanced payload generation (Shellcode, encoders).
* **Metasploit Framework:** The core engine for exploit development.

### Module D: Active Exploitation

* **Metasploit:** Execution of verified exploits.
* **SQLmap:** Automated SQL injection and database takeover.
* **Custom Scripts:** Python/Go scripts generated by the AI for specific CVEs.

### Module E: Privilege Escalation

* **PEASS Suite:**
  * *LinPEAS:* Linux Privilege Escalation Awesome Script.
  * *WinPEAS:* Windows Privilege Escalation Awesome Script.
* **Metasploit Local Exploits:** Post-exploitation modules for elevating privs.

### Module F: Post-Exploitation & Pivoting

* **Sliver:** Modern, cross-platform C2 framework (Replaces Empire). Automated via gRPC.
* **MITMproxy:** Intercepting and modifying traffic (Man-in-the-Middle).
* **Protocol Tunneling:**
  * *Chisel:* TCP/UDP tunneling over HTTP.
  * *Ligolo-ng:* Advanced tunneling interface (TUN/TAP).
* **Persistence:** Automated generation of persistence scripts (Cron, Registry, Services).

### Module G: Platform Security (Self-Protection)

* **AuthN/AuthZ:** JWT-based authentication with Role-Based Access Control (RBAC).
* **Audit Logs:** Immutable logs of all AI actions and user commands.
* **Encrypted Storage:** AES-256 encryption for sensitive findings (credentials, keys) in the database.

## 3. System Architecture

### Backend Stack

* **Language:** Python 3.11+
* **Framework:** **FastAPI** (Async support is critical for the Swarm).
* **Data Access Layer (DAL):**
  * **Repository Pattern:** Abstract all database interactions (CRUD) into repository classes (e.g., `TargetRepository`, `FindingRepository`).
  * **Dependency Injection:** Inject repositories into services using FastAPI's `Depends`. This decouples business logic from the DB, enabling easy testing with mock repositories.
* **Design Patterns:**
  * **Adapter Pattern:** Standardized interface (`ToolAdapter`) for all security tools.
  * **Strategy Pattern:** For scan profiles (e.g., `StealthStrategy`).
* **Task Queue:** **Arq** (Redis-based). Chosen over Celery for its native `asyncio` support, which aligns perfectly with FastAPI and the Swarm's non-blocking nature.
* **Real-Time Layer:** **WebSockets** (Socket.IO).
* **Database:**
  * **PostgreSQL:** Persistent storage.
  * **Redis:** Hot storage & Message Broker.
* **RAG Engine:**
  * **Vector Store:** Redis Stack (RediSearch).
  * **Data Sources:**
    * **CVE Database:** Ingested from local NVD JSON feeds.
    * **Tool Documentation:** Auto-generated by running tools with `--help` or `man` and indexing the output.
    * **Previous Findings:** Historical data from past assessments to identify recurring issues.
    * **Security Knowledge Base:** Curated list of attack patterns (CAPEC) and remediation guides.

### Containerization (Docker)

* **Development:** Uses `docker compose watch` for efficient hot-reloading of both backend (Python) and frontend (React) code.
* **Production:** Multi-stage builds to minimize image size.
* `spectra-app`: Main FastAPI backend.
* `spectra-worker`: **Arq** worker process. Runs inside the `spectra-tools` environment.
* `spectra-ai`: Local LLM inference engine.
* `spectra-db`: PostgreSQL.
* `spectra-redis`: Redis.
* `spectra-tools`: Base image with security tools.

## 4. Testing Strategy

To ensure reliability in a probabilistic AI system, we employ a multi-layered testing approach.

### 4.1. Test Philosophy

**Key Goals:**

* **No Mocking of Core Logic** (for E2E): Agents must actually call LLMs, Tools must actually run processes.
* **Real Targets**: E2E tests run against real, contained vulnerable targets (e.g., local Docker containers).
* **User Interaction**: Verify that user steering (changing scope, stopping, focusing) actually alters agent behavior.
* **Resilience**: Verify system behavior under network failure, LLM hallucinations, and tool crashes.
* **MAKER Framework Compliance**: Verify that the Mission Controller follows the MAKER framework and that the Consensus Engine actively votes on decisions via Quality Gates.

### 4.2. Test Layers

#### Unit Tests (Pytest)

* Test individual components (Parsers, Adapters, Agents) in isolation.
* **Mocking:** Use `MockLLMClient` to simulate AI responses deterministically.
* **Current Coverage:** 126 tests, 56% code coverage.
* **Key Test Files:**
  * `tests/test_ai.py` - Agent tests, LLM client tests, voting system tests
  * `tests/test_tools.py` - Tool registry, adapter, plugin safety tests
  * `tests/unit/test_mission.py` - Mission class, attack surface tracking
  * `tests/unit/test_consensus.py` - Quality gates, vote analysis
  * `tests/unit/test_security.py` - JWT, password hashing (100% coverage)

#### Integration Tests (Dockerized)

* Spin up a temporary `spectra-target` container (e.g., `tleemcjr/metasploitable2`).
* Run the full pipeline against this local target to verify tool execution and finding reporting.
* **Test Files:** `tests/integration/test_tools_container.py`

#### End-to-End (E2E) Tests

* Verify complete mission lifecycle with real LLM calls and real tool execution.
* **Test Files:** `tests/e2e/test_ui_flow.py`

### 4.3. E2E Test Scenarios

#### Scenario E2E-01: Core Workflow (Happy Path)

* **Trigger**: User starts a "Full Assessment" on `TEST_TARGET_IP`.
* **Expected Flow**:
    1. **Scope Agent**: Verifies target is reachable and within permission boundary.
    2. **Mission Controller**: Generates a `MissionPlan` with phases (Discovery → Enum → Vuln → Exploit).
    3. **Quality Gate (PLAN)**: Validates the proposed `MissionPlan` with 3 voters.
    4. **Tool Selector**: Picks `nmap` for Discovery, validated at TOOL_SELECTION gate.
    5. **Execution**: `nmap` runs, finds open ports (e.g., 80, 22).
    6. **Attack Surface**: `MissionManager` updates with discovered services.
    7. **Vector Generator**: Creates attack vectors (e.g., "Web Enumeration").
    8. **Exploitation**: `ExploitCrafter` attempts safe exploits, validated at PAYLOAD gate.
    9. **Reporting**: `ReporterAgent` generates a summary JSON/Markdown.
* **Success Criteria**: Mission status `completed`, findings list > 0, report generated.

#### Scenario E2E-02: User Steering

* **Trigger**: Start mission. Mid-way, inject steering command: "Focus on Web vulnerabilities only".
* **Expected Flow**:
    1. `MissionController` receives `SteeringAction`.
    2. Steering validated at REPLAN quality gate.
    3. Plan is updated: Network tasks deprioritized, Web tasks boosted.
    4. Subsequent tools are web-focused (e.g., `nikto`, `ffuf`).
* **Success Criteria**: Logs show steering received, plan updated, web-focused tools selected.

#### Scenario E2E-03: Emergency Stop

* **Trigger**: Start mission. Inject command: "STOP immediately".
* **Expected Flow**:
    1. `MissionManager` sets stop event.
    2. Running tools killed via `SIGKILL`.
    3. Mission status updates to `cancelled`.
* **Success Criteria**: Mission status is `cancelled` within 5 seconds.

#### Scenario E2E-04: Adaptive Retry on Failure

* **Trigger**: `ExploitCrafter` fails an attempt (e.g., "Connection Refused").
* **Expected Flow**:
    1. `MissionManager` captures failure, triggers `_handle_task_failure`.
    2. Replan validated at REPLAN quality gate.
    3. `ExploitCrafter` invoked with `previous_error`, suggests different approach.
    4. Retry loop executes with modified parameters.
* **Success Criteria**: At least one retry attempt with different parameters/tool.

#### Scenario E2E-05: Safety Blocking

* **Trigger**: Agent attempts to run `rm -rf /` or scan out-of-scope IP.
* **Expected Flow**:
    1. `SafetySupervisor` intercepts the `ToolExecutionRequest`.
    2. Action is BLOCKED before execution.
    3. Mission log records "[BLOCKED] Safety Violation".
* **Success Criteria**: Command *never* executes.

#### Scenario E2E-06: Consensus Rejection

* **Trigger**: Agent proposes action that fails quality gate validation.
* **Expected Flow**:
    1. Quality gate detects low confidence or risky action.
    2. Multiple LLM instances vote.
    3. If rejected, action is skipped or alternative is chosen.
* **Success Criteria**: Vote log shows rejection, alternative action taken.

## 5. User Interface (UI/UX)

**Design Language:** "Cinematic Data Density" (See `docs/design_system.md`)

* **Theme:** Deep Slate (`#020617`) with filmic noise texture and semantic glows.
* **Layout:** **Bento Grid** architecture. Everything is a tile.
* **Style:** Glassmorphism for overlays, sharp monospaced typography for data.
* **Responsiveness:** Fully mobile-responsive.
* **Tech Stack:** No complex build steps. Vanilla JavaScript (ES6+) or lightweight Alpine.js. All external libraries (Tailwind, Xterm.js, Cytoscape.js) loaded via CDN.

### Key Features

1. **Live "War Room" Dashboard:**
    * **Network Graph:** Interactive node graph (Cytoscape.js) showing discovered assets and their relationships.
    * **Terminal Streams:** **Xterm.js** windows showing raw output from tools like Nmap or Metasploit in real-time.
    * **Agent Chat:** A chat interface to talk to the "Swarm" (e.g., "Why did you choose Nikto?").
    * **Mission Control:** A command input for high-level directives (e.g., "Steer: Focus on persistence").
    * **Pentest Status:** Real-time progress bar and phase indicator (e.g., "Phase: Enumeration - 45% Complete").
    * **Steering Interface:** Buttons and input fields to pause, stop, or redirect the assessment mid-flight.

2. **Model Management UI:**
    * **Hot-Swapping:** Dropdown to switch models instantly (e.g., "Switch from Qwen-2.5-3B to DeepSeek-R1").
    * **Provider Toggle:** Switch between Local (Ollama) and Cloud (OpenAI/Anthropic) APIs.
    * **Resource Monitor:** Live CPU/GPU usage charts for the local models.

3. **Settings & Configuration:**
    * **Config Editor:** Visual editor for `config.py` settings (e.g., API keys, timeouts).
    * **Plugin Manager:** UI to enable/disable tool plugins and view their status.
    * **Auto-Setup Status:** View status of automated setup tasks (embeddings download, plugin installation) that ran on container launch.

4. **Target Management:**
    * **Asset DB:** CRUD interface for targets.
    * **Shell Handler:** A web-based reverse shell listener. If a target is owned, a terminal pops up in the browser connected to the session.

5. **Reporting Suite:**
    * **Live View:** HTML report with interactive charts (Chart.js).
    * **Exports:** One-click generation of PDF (Executive Summary), JSON (Raw Data), and Markdown (Developer Notes).

**UI Components:**

* **Icons:** FontAwesome (Free) for all UI elements.
* **Theme:** Dark mode default, consistent with "Cinematic Data Density".
