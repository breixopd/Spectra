Getting Started

Everything you need to know to start using {{ app_name }}.

Initial Setup

On first launch you'll be redirected to /setup. Create your admin account and configure your AI provider — either a local Ollama instance or a cloud API gateway (OpenAI, Anthropic, etc.) via TensorZero.

Create Your First Mission

From the Dashboard, enter a target IP or domain, add an optional directive describing your objective, and click Launch. You can also select a scan preset (Quick Recon, Full Assessment, Web App, Stealth, or Exploit Focus) for pre-configured workflows.

Understanding the Dashboard

The dashboard shows your active mission with real-time progress. The activity feed streams agent actions, tool outputs, and AI reasoning. The findings panel aggregates discovered vulnerabilities by severity — Critical, High, Medium, Low, and Info. Network topology is visualized in the graph view.

Reviewing Findings

Each finding includes severity, description, evidence, affected hosts, and remediation recommendations. Use the findings panel on the dashboard or navigate to mission details from History for a full breakdown.

Generating Reports

Go to Reports to generate PDF or JSON reports for any completed mission. Reports include executive summary, technical findings, evidence, and remediation priorities.

Manual Pentest Mode

Run security tools directly without AI orchestration.

Use Autonomous mode (Dashboard → Launch) for comprehensive, AI-driven assessments where the agents plan, execute, and adapt automatically. Use Manual mode when you want precise control — running specific tools with exact arguments, chaining scans in a custom pipeline, or investigating a specific finding deeper.

All installed plugins are available in manual mode. Common categories include:

Recon: nmap, subfinder, httpx, amass
Web: nikto, feroxbuster, sqlmap, wpscan
Brute: hydra, kerbrute, crackmapexec
Vuln: nuclei, searchsploit, testssl

See the Toolbox page for the full registry with install status and documentation links.

Navigate to Manual Mode → Execute tab. Select a tool from the catalog, fill in the target and parameters, then click Run. Output streams in real-time in the terminal panel below. You can run multiple tools concurrently in separate sessions.

Use the Pipeline tab to chain tools together. Output from one stage feeds into the next — for example, run nmap to discover open ports, then pipe those ports into nuclei for vulnerability scanning. Drag stages to reorder, and use the auto-parse feature to extract targets from previous output.

For full shell access, open a Web Shell session from any active sandbox. This gives you a terminal inside the tools container with all security tools pre-installed. For remote targets behind a VPN, upload your .ovpn configuration in Settings → VPN and select it when launching a mission.

API Access

Integrate {{ app_name }} into your workflow via the REST API.

Getting Your Token

Authenticate via POST /api/v1/auth/token with your username and password. The response includes a JWT access token.

curl -X POST '/api/v1/auth/token' \
  -d 'username=admin&password=yourpassword' \
  -H 'Content-Type: application/x-www-form-urlencoded'

Authentication

Include the token in every request as a Bearer token in the Authorization header:

Authorization: Bearer eyJhbGciOiJIUzI1NiIs...

Rate Limits & Best Practices

  • API calls are rate-limited per-user. Check response headers for limit info.
  • Use pagination for list endpoints (?page=1&per_page=20).
  • Subscribe to the WebSocket at /ws?token=JWT for real-time updates instead of polling.
  • All timestamps are in UTC ISO 8601 format.
See the full API Reference for all endpoints, schemas, and an interactive playground.

Services Management

Understanding and managing the platform's service topology.

{{ app_name }} can run services locally (Docker containers on the same host) or route them to external servers. Two gateway URLs control this: LLM_GATEWAY_URL for AI inference and SANDBOX_ORCHESTRATOR_URL for tool execution. When set, in-process implementations are transparently replaced by HTTP gateway clients. Configure these in Admin → Services or via environment variables.

Sandbox workers are isolated Docker containers (based on Kali Linux) where security tools execute. Each mission gets its own sandbox for isolation. Configure pool size, memory limits, CPU shares, and timeouts in Settings → Sandbox. For high-throughput deployments, use an external sandbox orchestrator to distribute work across multiple nodes.

Garage (or any S3-compatible endpoint) stores mission artifacts: scan outputs, evidence files, screenshots, and generated reports. Configure the endpoint URL, access key, secret key, and region in Settings. Without S3 configured, artifacts are stored locally on disk.

The Admin → Services tab shows real-time health status for all registered services with colored indicators: green (healthy), red (unhealthy), yellow (no health check), gray (disabled). The Observability page provides deeper metrics on system performance, queue depths, and resource usage.

Plugin System

Extend {{ app_name }} with custom tool integrations.

What Are Plugins?

Plugins are JSON configuration files that describe how to install and run a security tool. Each plugin defines the tool's name, description, installation commands, execution syntax, argument schema, and output parsing rules. Plugins live in the plugins/ directory.

Sandbox Execution

All tools execute inside isolated sandbox containers — never on the host. Each sandbox has its own filesystem, network namespace, and resource limits. When Plugin Safe Mode is enabled in Settings, only digitally signed plugins are allowed to run.

Adding Plugins

To add a new plugin, create a JSON file in the plugins/ directory or use the Toolbox → Upload feature. The plugin auto-installs in the tools container on next startup. Use Toolbox → Create for an interactive plugin builder.

Troubleshooting

Common issues and how to resolve them.

  • Check the Agents (Overseer) page for agent status and logs.
  • Verify the AI provider is connected in Settings — use the "Test Default Profile" button.
  • Ensure the tools container is running (check Admin → Services).
  • Look at the Observability page for queue backlogs or errors.
  • If fully automated mode is off, check for pending human approval requests.
  • For Ollama: ensure the Ollama service is running and accessible at the configured host. Verify the model is pulled (ollama pull model_name).
  • For Cloud API: check your API key is valid and base URL is correct. Some providers require specific URL formats.
  • Use the Test button in Settings to verify connectivity before launching missions.
  • Check logs in Observability for specific error messages from the LLM client.
  • "Docker not accessible" — ensure Docker is running and the socket is mounted (/var/run/docker.sock).
  • Container OOM killed — increase memory limits in Settings → Sandbox. Enable OOM escalation for automatic retry with higher limits.
  • Pool exhausted — increase max containers or wait for current containers to complete. Check for leaked sessions.
  • For external orchestrator issues, verify the SANDBOX_ORCHESTRATOR_URL is reachable and the remote service is healthy.
  • 403 Forbidden — your user role may lack the required permission. {{ app_name }} uses RBAC with admin, staff, and user roles. Contact your admin to adjust your role.
  • 401 Unauthorized — your session may have expired. Log out and log in again to refresh your token.
  • Admin pages not visible — only users with the admin or staff role can see the Admin nav link. It appears automatically when the API confirms your role.
